# Git Workflow

Этот репозиторий использует простой trunk-based workflow: `main` всегда
зелёный, а вся работа идёт через короткоживущие ветки и PR.

## Базовые ветки

### `main`

Назначение: всегда зелёная ветка. Проект компилируется, тесты проходят,
`doctor` работает. С неё ставятся теги.

Жизнь: вечная.

Правило: не пушить фичи напрямую в `main`. Изменения попадают через PR.

### `feature/<scope>`

Назначение: одна фича или одна секция спеки.

Примеры:

- `feature/wire-codec-anthropic`
- `feature/codegraph-watchdog`
- `feature/router-planner`

Жизнь: дни-недели. Удаляется после merge.

Куда мёрджится: `main` через PR.

### `fix/<scope>`

Назначение: точечные некритические баги.

Пример:

- `fix/typer-help-color`

Жизнь: часы-дни.

Куда мёрджится: `main` через PR.

### `docs/<topic>`

Назначение: только документация: README, спека, переводы, workflow docs.

Примеры:

- `docs/ru-readme-update`
- `docs/spec-section-26`
- `docs/git-workflow`

Жизнь: часы-дни.

Куда мёрджится: `main` через PR.

### `chore/<thing>`

Назначение: tooling, dependencies, CI, конфиги.

Примеры:

- `chore/ruff-upgrade`
- `chore/ci-mypy-strict`

Жизнь: часы-дни.

Куда мёрджится: `main` через PR.

## Расширенные ветки

### `phase/<N>-<name>`

Использовать, когда работа над фазой спеки идёт больше недели и состоит из
нескольких feature-веток.

Пример:

- `phase/4-codegraph`

Задача: собрать несколько связанных feature-веток и потом открыть один PR в
`main`.

### `release/v0.X`

Использовать при подготовке версии: freeze, только bugfix. С release-ветки
ставится тег.

Пример:

- `release/v0.5`

### `hotfix/<issue>`

Использовать для срочного патча уже выпущенной версии. Ветка создаётся от
тега, мёрджится в `main` и обратно в активную `release/*`.

### `experiment/<name>`

Использовать для research-кода, который может не попасть в продукт.

Пример:

- `experiment/tree-sitter-vs-libcst`

### `draft/<topic>`

Использовать для долгих design-exploration без давления merge.

Пример:

- `draft/voice-input-spec`

### `gh-pages`

Использовать только когда появится docs-сайт для GitHub Pages.

## Что не заводить отдельной веткой

- Не заводить ветки версий вроде `v0.4.0`. Версии это теги:
  `git tag -a v0.4.0 -m "..."`, затем `git push --tags`.
- Не заводить `dev` или `develop` для текущего соло-формата.
- Не заводить `personal/whipedev/*`.

## Правило именования

Формат:

```text
<type>/<kebab-case-scope>
```

Только lowercase, цифры и hyphen. Без пробелов, заглавных букв и emoji.

## PR hygiene

1. Делать self-PR даже для соло-работы.
2. `main` должен быть защищён от force-push и прямого push.
3. Для feature-веток использовать squash & merge.
4. Удалять слитые ветки автоматически.
5. Использовать Conventional Commits для PR title и итогового squash commit:
   `feat:`, `fix:`, `docs:`, `chore:`.

## Agent rule

Любой агент, работающий в этом репозитории, должен:

1. Проверить текущую ветку перед изменениями.
2. Если текущая ветка `main`, создать ветку подходящего типа.
3. Не пушить изменения напрямую в `main`.
4. Перед PR или merge прогнать проверки проекта.
5. Описывать коммит подробно: `What changed`, `Why`, `Validation`.
